Github crashcourse for university students

Shopping style

This short, no bullshit crash course is aimed at computer science university students on how to use github.

Why use github at all?

  • Really fast and easily accessible on all of your devices as well as ease of sharing
  • Easy to organise and oversee your code
  • Ability to revert back any changes
  • An industry standard that will come in handy in the long run

Alright, so where do I start?

Head on over to the git download site and follow the instructions for your OS. When installing there should be no need to change any of the default settings.

So I have git now, where do I start?

Before we start, let's draw a really quick image of how git and github will work in our case. Imagine a store(head), a storage/warehouse(index) and a laboratory(working directory). You are currently working in the laboratory developing something new.
We will try to build up this store and operate it.

To create a this, navigate to the working directory, open a terminal, and type:

git init

Note: A git repository is essentially the hidden .git/ map in your working directory. To delete the repository, just delete that folder.

With this, our barebone store looks like this.

So I have developed something, how do I "save"?

To transfer your things to the index(storage), you will need to add them to it with the following command: 

git add *

Note: You can use the learnt formatting from computer systems to specify which items to add to the staging area.
In this case, * means that we add everything in the working directory to the index(storage).

Once you have fully stocked up the index(storage), we need to create a commit(inventory) of it, so we know what to put out into the actual head(store). These commits(inventories) will all get saved, and you can easily access information even about past commits(inventories).

git commit -m "Commit message"

Great, what about the head(store)?

Head over to github and register for an account and create a repository.

With this we have created the barebones of the head(store). But there is no connection between the head(store) and our index(warehouse).
We do this with the following command:

git remote add origin "git repository link"

Where you should replace "git repository link" with the url of your github repo.

Now that the two are connected, we can finally push our commit(inventory) to the head(store) for display.

git push origin master

This is the formal full version, you could also just use git push normally.

And with that, you have published your first commit to your github repo. You can view your code online at your repo's github page.

But what happens when I go home and want to use the code there?

To create a second index(warehouse), you could use git init on your home PC. However, that repository would be empty. Instead, to create one from the head(store), use:

git clone "git repository url"

You can now use the working directory as usual.

Also note that you only need to use clone for the initial creation.

Afterwards, whenever you log into a new location, to update all of the files to the latest commit(inventory), use:

git pull

Can I do something about all the junk files in my project?

A lot of times, there is no need to sync some junk files. For example IDE generated files, object and executable files etc. You can create a file that tells git to ignore certain files when you add files to the index.
This is the .gitignore. To create one, use the following command in the workign folder:

touch .gitignore

You can type filenames in this file that you want to ignore. You can use all the formatting learned during computer systems.
An example to ignore all object and executable files would be: 

*.o
                    
*.exe

List of commands in the order of the pipeline

Command Description Usage
Initializing
Git init Initializes an empty git in the working directory. Use at the very beginning to create a new git repo from your existing code.
Git clone "github repo url" Clones a github from a remote path. Use this when you first want to get your project to a new device.
Git pull Fetches and then merges the current working branch. Use this to get the latest commit/update when you log into a previously used device.
Adding and removing
Git status Writes out the status of the index (tracked files, changed but untracked files). Use this to ensure you have the right items in the index.
Git add "filename" Adds the item(s) to the index. Use this to specify which files the next commit should track.
Git rm "filename" Removes the item(s) to the index. (use --cached to remove items already in the index.) Use this to delete a file that will be tracked in the next commit.
Committing and synchronizing
Git commit -m "Commit message" Creates a new commit from the current state of the index. Use this to create a "save point" every time you have made some changes. Don't forget to add your files.
Git remote add origin "github repo url" Connects the remote to the local repo. Use this to create the initial connection between the github repo and your local. Only need to do this once.
Git push Pushes your current branch(no need to worry about this for now) to your remote repo. Use this to "upload" all of your changes online. Don't forget to push when you finish your tutorials.
